為什麼 Markdown 原本勝出？

簡單、可讀、可編輯

標題、列表、程式碼區塊與表格，足以支撐大多數 AI 文字輸出。

Token 效率高

早期 GPT-4 8,192 token 限制下，Markdown 比 HTML 更省輸出與上下文成本。

Git 與知識庫友善

純文字 diff、版本控制、Obsidian / GitHub / docs pipeline 都成熟。

人機都能處理

模型容易生成，人類也能直接修，讓它成為 agent 工作流的共同語言。

Thariq 的反轉：AI 輸出變太複雜

Markdown 文字牆

  • 長規格容易掃過不讀
  • 多方案比較只能線性排列
  • 圖表常退化成 ASCII art
  • PR review 缺少視覺層級
  • 難做決策輸入與回傳
VS

HTML artifact

  • 可用版面引導閱讀
  • 並排、卡片、timeline、tabs
  • SVG、圖表、互動元件
  • diff annotation 與 severity color
  • button / slider / export 回 agent

核心不是「HTML 比 Markdown 高級」，而是 AI 產物正在從 report 轉向 interface。

HTML 的 5 個優勢

資訊密度

CSS、SVG、表格、圖片、canvas 與結構化區塊能把多維資訊放在同一頁。

視覺清晰

使用色彩、間距、導覽與響應式版面，降低人類閱讀大型 plan 的負擔。

容易分享

上傳後就是 URL，非技術同事不需要 Markdown viewer。

雙向互動

slider、toggle、拖拉卡片與 copy button 可把人的決策輸出回 prompt。

設計與原型

HTML 可直接呈現 mockup、component variants、動畫參數與設計系統 token。

讓人留在迴圈中

Thariq 的真正理由是：HTML 讓他更願意審閱 agent 的選擇。

反方觀點：HTML 不是免費午餐

Token 與成本

反方估計 HTML 可能需要 2 到 4 倍輸出 token，生成更慢，輸出 token 也更貴。

Diff 與維護

大型 HTML 檔較難人工 diff、review、局部修改；Markdown 仍是長期紀錄的強項。

安全與信任

HTML 可能含 JavaScript；不可信的 AI 生成檔需要 sandbox、審查或禁用腳本。

真正問題可能是輸出膨脹

Reddit 討論指出：如果 100 行 Markdown 沒人讀，可能該先要求 agent 精簡。

最佳分工：Markdown for memory, HTML for review

Markdown 適合

  • 長期知識庫與 repo docs
  • AGENTS.md、CLAUDE.md、README
  • 版本控制與文字 diff
  • 人類與 agent 反覆編輯
  • 低成本中間素材
+

HTML 適合

  • 簡報、報告、教學頁
  • PR review 與設計審閱
  • 多方案比較與決策介面
  • 互動式調參與 micro-editor
  • 非技術同事瀏覽與分享
Source of truth: Markdown Human review layer: HTML Structured data: JSON/YAML Publish target: browser

做簡報時可以採用的敘事

1. 舊世界
AI 主要產生文字答案，Markdown 夠輕、夠快、夠可讀。
2. 轉折
Coding agents 開始產生長規格、PR review、設計方案與研究報告，人類不再細讀。
3. 新解法
HTML 把輸出變成可瀏覽、可互動、可分享的工作介面。
4. 平衡
HTML 不取代所有 Markdown，而是接手「要給人審閱和決策」的那一層。

Final takeaway

AI 時代的文件，正在變成介面

不要問「Markdown 還是 HTML 誰贏」。先問：這份輸出是要被保存、被編輯，還是要被人看懂並做決策？
